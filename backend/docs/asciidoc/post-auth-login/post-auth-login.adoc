= Метод для аутентификации пользователя и получения JWT
:toc-title: Содержание
:table-stripes: even
:table-caption!:

== 1. Назначение
Метод выполняет аутентификацию пользователя по паре `email`/`password`. В случае успешной проверки возвращает подписанный JWT, который клиент должен использовать в заголовке `Authorization: Bearer <token>` для всех защищённых эндпоинтов.

Токен содержит клеймы `user_id`, `role`, `exp` (Unix timestamp), подписан алгоритмом HS256 секретом из конфига сервиса и живёт 30 дней с момента выдачи.

Метод публичный (без авторизации). На все случаи, когда аутентификация не прошла (пользователь не найден, пароль не совпадает), возвращается единая ошибка 401 с обезличенным сообщением — это сделано намеренно, чтобы не давать возможности определить, зарегистрирован ли email в системе.

=== 4.1. Валидация входных параметров
. Попытаться декодировать тело запроса как JSON в структуру `LoginRequest` с полями `email`, `password`. Если JSON некорректен — вернуть исключение с http code 400 (Bad Request) и сообщение: "некорректное тело запроса". Завершить работу.
. Проверить:
+
|===
|Параметр |Условие |Результат

|`email`, `password`
|Хотя бы одно поле пустое
|Вернуть исключение с http code 400 (Bad Request) и сообщение: "email и пароль обязательны"
|===

=== 4.2. Запрос к БД на получение пользователя по email
. В таблице `users` найти запись, у которой `email = $email`, и подтянуть поля `id`, `email`, `password_hash`, `role`:
.. если запись не найдена — вернуть исключение с http code 401 (Unauthorized) и сообщение: "неверный email или пароль". Завершить работу;
.. если произошла ошибка БД — вернуть исключение с http code 500 (Internal Server Error) и сообщение: "внутренняя ошибка сервера, попробуйте позже";
.. иначе — перейти к шагу 4.3.

=== 4.3. Сравнение пароля с хэшем
. Выполнить `bcrypt.CompareHashAndPassword(password_hash, password)`:
.. если функция вернула ошибку (пароль не совпал) — вернуть исключение с http code 401 (Unauthorized) и сообщение: "неверный email или пароль". Завершить работу;
.. иначе — перейти к шагу 4.4.

=== 4.4. Формирование и подписание JWT
. Вычислить время истечения: `expiresAt = now() + 30 дней`.
. Собрать payload:
+
[source, JSON]
----
{
  "user_id": "<user.ID>",
  "role":    "<user.Role>",
  "exp":     <expiresAt.Unix()>
}
----
. Подписать токен алгоритмом HS256 секретом `jwtSecret` из конфига сервиса.
. Если подписание вернуло ошибку — вернуть исключение с http code 500 (Internal Server Error) и сообщение: "внутренняя ошибка сервера, попробуйте позже".
. Перейти к шагу 4.5.

=== 4.5. Формирование итогового ответа
. Собрать тело ответа:
.. `access_token` — подписанный JWT;
.. `token_type` — `"Bearer"`;
.. `expires_at` — время истечения в формате ISO-8601.
. Перейти к шагу 4.6.

=== 4.6. Отправка ответа об успехе
. Отправить сформированное тело с кодом успеха 200 OK.
. Завершить работу.
