= Метод для получения профиля текущего пользователя
:toc-title: Содержание
:table-stripes: even
:table-caption!:

== 1. Назначение
Метод возвращает профиль пользователя, связанного с переданным JWT. Идентификатор пользователя извлекается из клейма `user_id`, сам запрос к БД выполняется по этому идентификатору.

В ответ включаются все поля записи в таблице `users` за исключением `password_hash` (поле помечено тегом `json:"-"` в модели и не сериализуется).

Метод требует авторизации. Используется фронтендом при инициализации пользовательского контекста после логина, а также при открытии страницы профиля.

== 2. Входные параметры
|===

|Authorization
|header
|string
|Заголовок авторизации формата `Bearer <jwt>`. Токен должен быть валидным и не истёкшим.

|===

Тело запроса не передаётся.

|===
|Параметр |Формат |Описание |Примеры

|id
|string
|Идентификатор пользователя (UUID)
|"9d3c5b2a-6e8f-4a1c-b0d2-7e9f1a3b5c6d"

|email
|string
|Электронная почта пользователя
|"user@example.com"

|optional string
|ФИО пользователя (может быть null, если не заполнено)
|"Андрей Иванов"

|phone
|optional string
|Номер телефона
|"+7 999 123-45-67"

|optional string
|Ссылка/юзернейм Telegram

|optional string

|role
|string
|Роль пользователя. Допустимые значения: `"user"`, `"admin"`.

|datetime
|Время создания записи в БД

=== 4.1. Идентификация пользователя по JWT
. Извлечь заголовок `Authorization` из запроса:
.. если заголовок отсутствует — вернуть исключение с http code 401 (Unauthorized) и сообщение: "отсутствует токен авторизации";
.. если формат заголовка не соответствует шаблону `Bearer <token>` — вернуть исключение с http code 401 (Unauthorized) и сообщение: "неверный формат заголовка Authorization".
. Распарсить JWT и проверить подпись секретом сервиса. Проверить, что алгоритм подписи принадлежит семейству HMAC:
.. если алгоритм не HMAC — вернуть исключение с http code 401 (Unauthorized) и сообщение: "неподдерживаемый алгоритм подписи токена";
.. если подпись невалидна или токен истёк — вернуть исключение с http code 401 (Unauthorized) и сообщение: "невалидный или просроченный токен".
. Извлечь из клеймов `user_id`, `role`:
.. если `user_id` пустой — вернуть исключение с http code 401 (Unauthorized) и сообщение: "невалидный токен: отсутствует user_id".
. Положить `user_id` и `role` в контекст запроса.

=== 4.2. Валидация входных параметров
. Проверить, что HTTP-метод запроса — `GET`. Если нет — вернуть исключение с http code 405 (Method Not Allowed) и сообщение: "метод не поддерживается".
. Дополнительные параметры запроса (query/body) не обрабатываются.

=== 4.3. Запрос к БД на получение пользователя
. В таблице `users` найти запись, у которой `id = $user_id`, и подтянуть поля `id`, `email`, `password_hash`, `full_name`, `phone`, `tg_link`, `vk_link`, `role`, `created_at`:
.. если запись не найдена (токен валиден, но пользователь был удалён) — вернуть исключение с http code 404 (Not Found) и сообщение: "пользователь не найден";
.. если произошла ошибка БД — вернуть исключение с http code 500 (Internal Server Error) и сообщение: "внутренняя ошибка сервера, попробуйте позже";
.. иначе — перейти к шагу 4.4.

=== 4.4. Формирование итогового ответа
. Собрать тело ответа из полученной записи: `id`, `email`, `full_name`, `phone`, `tg_link`, `vk_link`, `role`, `created_at`. Поле `password_hash` не включается (исключено из JSON-сериализации на уровне модели).
. Перейти к шагу 4.5.

=== 4.5. Отправка ответа об успехе
. Отправить сформированное тело с кодом успеха 200 OK.
. Завершить работу.
