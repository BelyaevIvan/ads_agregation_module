= Метод для обновления профиля текущего пользователя
:toc-title: Содержание
:table-stripes: even
:table-caption!:

== 1. Назначение
Метод выполняет частичное обновление профиля пользователя, связанного с переданным JWT. Поддерживает обновление четырёх полей: `full_name`, `phone`, `tg_link`, `vk_link`.

Обновление выполнено через `COALESCE` на уровне SQL: переданные в теле `null`-значения игнорируются и сохраняют в БД прежнее значение. Это означает, что клиент может отправлять только те поля, которые действительно нужно изменить, — остальные можно не включать в тело.

Поля `email`, `password_hash`, `role`, `created_at` через этот эндпоинт изменить нельзя.

Метод требует авторизации.

=== 4.1. Идентификация пользователя по JWT
. Извлечь заголовок `Authorization` из запроса:
.. если заголовок отсутствует — вернуть исключение с http code 401 (Unauthorized) и сообщение: "отсутствует токен авторизации";
.. если формат заголовка не соответствует шаблону `Bearer <token>` — вернуть исключение с http code 401 (Unauthorized) и сообщение: "неверный формат заголовка Authorization".
. Распарсить JWT и проверить подпись секретом сервиса. Проверить, что алгоритм подписи принадлежит семейству HMAC:
.. если алгоритм не HMAC — вернуть исключение с http code 401 (Unauthorized) и сообщение: "неподдерживаемый алгоритм подписи токена";
.. если подпись невалидна или токен истёк — вернуть исключение с http code 401 (Unauthorized) и сообщение: "невалидный или просроченный токен".
. Извлечь из клеймов `user_id`:
.. если `user_id` пустой — вернуть исключение с http code 401 (Unauthorized) и сообщение: "невалидный токен: отсутствует user_id".
. Положить `user_id` в контекст запроса.

=== 4.2. Валидация входных параметров
. Проверить, что HTTP-метод запроса — `PUT`. Если нет — вернуть исключение с http code 405 (Method Not Allowed) и сообщение: "метод не поддерживается".
. Попытаться декодировать тело запроса как JSON в структуру `UpdateProfileRequest` с полями `full_name`, `phone`, `tg_link`, `vk_link` (все поля — указатели на строку, `null` допустим). Если JSON некорректен — вернуть исключение с http code 400 (Bad Request) и сообщение: "некорректное тело запроса".
. Выполнить проверки длин:
+
|===
|Параметр |Условие |Результат

|`full_name`
|Передан и длина превышает 255 символов
|Вернуть исключение с http code 400 (Bad Request) и сообщение: "full_name не может превышать 255 символов"

|`phone`
|Передан и длина превышает 30 символов
|Вернуть исключение с http code 400 (Bad Request) и сообщение: "phone не может превышать 30 символов"

|`tg_link`
|Передан и длина превышает 255 символов
|Вернуть исключение с http code 400 (Bad Request) и сообщение: "tg_link не может превышать 255 символов"

|`vk_link`
|Передан и длина превышает 255 символов
|Вернуть исключение с http code 400 (Bad Request) и сообщение: "vk_link не может превышать 255 символов"
|===

=== 4.3. Запрос к БД на обновление профиля
. Выполнить запрос с `COALESCE` — переданные `null`-поля сохраняют прежние значения в БД:
+
[source, sql]
----
UPDATE users SET
    full_name = COALESCE($2, full_name),
    phone     = COALESCE($3, phone),
    tg_link   = COALESCE($4, tg_link),
    vk_link   = COALESCE($5, vk_link)
WHERE id = $1
RETURNING id, email, password_hash, full_name, phone, tg_link, vk_link, role, created_at
----
. Параметры запроса: `$1 = user_id` из JWT, `$2...$5` — значения из тела запроса.
. Если запрос не вернул ни одной строки (пользователь удалён из БД) — вернуть исключение с http code 404 (Not Found) и сообщение: "пользователь не найден".
. Если БД вернула ошибку — вернуть исключение с http code 500 (Internal Server Error) и сообщение: "внутренняя ошибка сервера, попробуйте позже".
. Перейти к шагу 4.4.

=== 4.4. Формирование итогового ответа
. Собрать тело ответа из обновлённой записи: `id`, `email`, `full_name`, `phone`, `tg_link`, `vk_link`, `role`, `created_at`. Поле `password_hash` не включается.
. Перейти к шагу 4.5.

=== 4.5. Отправка ответа об успехе
. Отправить сформированное тело с кодом успеха 200 OK.
. Завершить работу.
