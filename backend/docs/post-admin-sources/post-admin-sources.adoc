= Метод для добавления нового источника мониторинга
:toc-title: Содержание
:table-stripes: even
:table-caption!:

== 1. Назначение
Метод позволяет администратору добавить новый источник мониторинга (Telegram-канал или VK-группу) в систему. После добавления запись попадает в таблицу `sources` с флагом `is_active = TRUE`, и фоновый сервис `monitor-service` начнёт обрабатывать её при следующей итерации опроса.

Метод доступен только пользователям с ролью `admin`.

== 2. Входные параметры
Входные параметры передаются в теле запроса в формате JSON.

|===
|Параметр |Положение |Формат |Обязательный |Описание |Примеры

|platform
|body
|string
|да
|Платформа источника. Допустимые значения: `"telegram"` или `"vk"`
|"telegram", "vk"

|external_id
|body
|string
|да
|ID канала/группы во внешней системе
|"-1001234567890", "123456789"

|title
|body
|optional string
|нет
|Отображаемое название (канал/группа)
|"@sneakers_moscow"
|===

Требуется заголовок `Authorization: Bearer <jwt>` с валидным токеном пользователя, имеющего роль `admin`.

=== 4.1. Идентификация пользователя по JWT
. Извлечь заголовок `Authorization` и распарсить JWT.
.. если токен отсутствует, невалиден или просрочен — вернуть исключение с http code 401 (Unauthorized) и сообщение: "невалидный или просроченный токен". Завершить работу;
.. иначе — извлечь `user_id` и `role` из claims, перейти к шагу 4.2.

=== 4.2. Проверка роли администратора
. Проверить, что значение `role` из JWT равно `"admin"`.
.. если роль отличается — вернуть исключение с http code 403 (Forbidden) и сообщение: "доступ запрещён: требуется роль администратора". Завершить работу;
.. иначе — перейти к шагу 4.3.

=== 4.3. Валидация входных параметров
|===
|Параметр |Условие |Результат

|тело запроса
|Не удалось распарсить JSON
|Вернуть исключение с http code 400 (Bad Request) и сообщение: "некорректное тело запроса"

|`platform`
|Значение отличается от `"telegram"` и `"vk"`
|Вернуть исключение с http code 400 (Bad Request) и сообщение: "platform должен быть 'telegram' или 'vk'"

|`external_id`
|Пустая строка
|Вернуть исключение с http code 400 (Bad Request) и сообщение: "external_id обязателен"
|===

=== 4.4. Проверка уникальности пары (platform, external_id)
. Выполнить запрос:
+
[source, SQL]
....
SELECT EXISTS(SELECT 1 FROM sources WHERE platform = $1 AND external_id = $2)
....
. Если запрос завершился ошибкой — вернуть исключение с http code 500 (Internal Server Error). Завершить работу.
. Если источник с такой парой уже существует — вернуть исключение с http code 409 (Conflict) и сообщение: "источник с такой комбинацией platform/external_id уже существует". Завершить работу.
. Иначе — перейти к шагу 4.5.

=== 4.5. Создание записи в БД
. Выполнить запрос:
+
[source, SQL]
....
INSERT INTO sources (platform, external_id, title) VALUES ($1, $2, $3)
RETURNING id, platform, external_id, title, is_active, added_at
....
. Поле `is_active` заполняется значением по умолчанию `TRUE` на стороне БД; `added_at` — текущим временем.
. Если запрос завершился ошибкой — вернуть исключение с http code 500 (Internal Server Error). Завершить работу.
. Перейти к шагу 4.6.

=== 4.6. Формирование итогового ответа
. Собрать тело ответа из возвращённых БД полей: `id`, `platform`, `external_id`, `title`, `is_active`, `added_at`.
. Перейти к шагу 4.7.

=== 4.7. Отправка ответа об успехе
. Отправить ответ, сформированный на шаге 4.6, с кодом успеха 201 Created.
. Завершить работу.
