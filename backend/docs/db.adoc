== Описание хранилища
СУБД: **PostgreSQL 15+**

Наименование базы данных:

* Тестовый стенд — `brandhunt`
* Продуктивный стенд — `brandhunt`

Доступ к базе данных осуществляется под пользователями:

* Тестовый стенд — `brandhunt`
* Продуктивный стенд — `brandhunt`

Данные хранятся в базе данных в виде таблиц.

[cols="30%, 70%"]
|===
|Таблица |Описание

|<<users, users>>
|Пользователи системы

|<<sources, sources>>
|Источники мониторинга (Telegram-каналы, VK-группы)

|<<listings, listings>>
|Объявления о продаже товаров

|<<listing_photos, listing_photos>>
|Фотографии объявлений

|<<favorites, favorites>>
|Избранные объявления пользователей

|<<monitoring_log, monitoring_log>>
|Технический лог обработки сообщений мониторами
|===

== Таблицы
[[users]]
=== 1. Пользователи
Наименование: `users` +
Описание: Таблица пользователей системы
[cols="20%, 15%, 15%, 25%, 25%"]
|===
|Параметр |Тип данных |Доп. ограничения |Описание |Примеры

|id
|UUID
|PRIMARY KEY, NOT NULL, DEFAULT gen_random_uuid()
|Идентификатор пользователя
|"5a7f3c8e-1b2d-4e5f-8a9b-0c1d2e3f4a5b"

|email
|VARCHAR(255)
|NOT NULL, UNIQUE
|Электронная почта (уникальный логин)
|"user@example.com"

|password_hash
|TEXT
|NOT NULL
|bcrypt-хэш пароля (cost 12)
|"$2a$12$..."

|full_name
|VARCHAR(255)
|—
|ФИО пользователя
|"Иван Иванов"

|phone
|VARCHAR(30)
|—
|Номер телефона
|"+7 999 123-45-67"

|tg_link
|VARCHAR(255)
|—
|Ссылка или юзернейм Telegram
|"@andrey_tg"

|vk_link
|VARCHAR(255)
|—
|Ссылка ВКонтакте
|"vk.com/andrey"

|role
|VARCHAR(10)
|NOT NULL, DEFAULT 'user'
|Роль пользователя. Только одно значение
|"user", "admin"

|created_at
|TIMESTAMP WITH TIME ZONE
|NOT NULL, DEFAULT NOW()
|Дата регистрации
|2026-04-17 12:05:00+00
|===

[[sources]]
=== 2. Источники мониторинга
Наименование: `sources` +
Описание: Таблица источников мониторинга — Telegram-каналы и группы ВКонтакте, из которых сервис забирает новые посты
[cols="20%, 15%, 15%, 25%, 25%"]
|===
|Параметр |Тип данных |Доп. ограничения |Описание |Примеры

|id
|UUID
|PRIMARY KEY, NOT NULL, DEFAULT gen_random_uuid()
|Идентификатор источника
|"b2c3d4e5-a7f8-4b9c-8d0e-1f2a3b4c5d6e"

|platform
|VARCHAR(10)
|NOT NULL
|Платформа источника. Только одно значение
|"telegram", "vk"

|external_id
|VARCHAR(255)
|NOT NULL
|Идентификатор канала или группы во внешней системе (chat_id для TG, group_id для VK)
|"-1001234567890", "-225463035"

|title
|VARCHAR(255)
|—
|Человекочитаемое название канала/группы
|"@sneakers_moscow", "Арбузик"

|is_active
|BOOLEAN
|NOT NULL, DEFAULT TRUE
|Включён ли мониторинг этого источника. Отключённые источники остаются в БД, но не опрашиваются воркерами
|true, false

|added_at
|TIMESTAMP WITH TIME ZONE
|NOT NULL, DEFAULT NOW()
|Дата добавления источника
|2026-04-17 12:05:00+00
|===

На таблицу наложено ограничение уникальности: `UNIQUE (platform, external_id)` — один и тот же канал/группу нельзя добавить дважды.

[[listings]]
=== 3. Объявления
Наименование: `listings` +
Описание: Таблица объявлений о продаже товаров. Одна запись — один товар. Если в одном посте LLM извлекла несколько товаров, для каждого создаётся отдельная запись с общим `source_id`, `original_text`, `post_url`, `posted_at`
[cols="18%, 15%, 15%, 25%, 27%"]
|===
|Параметр |Тип данных |Доп. ограничения |Описание |Примеры

|id
|UUID
|PRIMARY KEY, NOT NULL, DEFAULT gen_random_uuid()
|Идентификатор объявления
|"5a7f3c8e-1b2d-4e5f-8a9b-0c1d2e3f4a5b"

|source_id
|UUID
|FOREIGN KEY → `sources(id)` ON DELETE SET NULL
|Идентификатор источника, из которого пришёл пост. Может быть `NULL`, если источник удалён
|"b2c3d4e5-a7f8-4b9c-8d0e-1f2a3b4c5d6e"

|original_text
|TEXT
|—
|Оригинальный текст поста (единый для всех товаров из одного поста)
|"Продаю Nike Air Max 90 белые 42 размер..."

|post_url
|TEXT
|NOT NULL
|Ссылка на оригинальный пост в TG/VK
|"https://t.me/sneakers_moscow/123"

|posted_at
|TIMESTAMP WITH TIME ZONE
|—
|Дата публикации поста в источнике
|2026-04-17 12:00:00+00

|brand
|VARCHAR(100)
|—
|Бренд товара, извлечённый LLM
|"Nike", "Stone Island"

|model
|VARCHAR(255)
|—
|Модель товара, извлечённая LLM
|"Air Max 90"

|category
|VARCHAR(100)
|—
|Категория товара. Только одно значение из ограниченного набора
|"Одежда", "Обувь", "Электроника", "Аксессуары"

|color
|VARCHAR(100)
|—
|Цвет товара
|"White"

|price
|NUMERIC(12,2)
|—
|Цена в рублях. `NUMERIC` выбран вместо `FLOAT` во избежание погрешностей арифметики
|8500.00

|city
|VARCHAR(100)
|—
|Город продавца
|"Москва"

|condition
|VARCHAR(50)
|—
|Состояние товара. Только одно значение
|"new", "used"

|size_rus
|TEXT[]
|—
|Размеры в российской размерной сетке. Массив, т.к. в одном объявлении может быть несколько размеров
|{"42","43","44"}

|size_us
|TEXT[]
|—
|Размеры в американской размерной сетке
|{"8.5","9","9.5"}

|size_eu
|TEXT[]
|—
|Размеры в европейской размерной сетке
|{"42","43"}

|is_hidden
|BOOLEAN
|NOT NULL, DEFAULT FALSE
|Признак, что объявление скрыто администратором из публичной выдачи. Скрытые объявления остаются в БД, видны только через админские эндпоинты
|true, false

|llm_raw
|JSONB
|—
|Сырой JSON-ответ LLM для конкретного товара. Хранится для отладки и возможности переразбора без повторного вызова LLM
a|
.Раскрыть пример
[%collapsible]
====
[source, JSON]
....
{
    "brand": "Nike",
    "model": "Air Max 90",
    "category": "Обувь",
    "color": "White",
    "price": 8500,
    "city": "Москва",
    "condition": "new",
    "size_rus": ["42", "43"],
    "size_eu": ["42"],
    "size_us": ["8.5"]
}
....
====

|created_at
|TIMESTAMP WITH TIME ZONE
|NOT NULL, DEFAULT NOW()
|Дата создания записи в БД (момент, когда монитор сохранил объявление)
|2026-04-17 12:05:00+00
|===

[[listing_photos]]
=== 4. Фотографии объявлений
Наименование: `listing_photos` +
Описание: Таблица фотографий, привязанных к объявлениям. Физические файлы лежат в MinIO, здесь хранятся только URL и метаданные
[cols="18%, 15%, 15%, 25%, 27%"]
|===
|Параметр |Тип данных |Доп. ограничения |Описание |Примеры

|id
|UUID
|PRIMARY KEY, NOT NULL, DEFAULT gen_random_uuid()
|Идентификатор фотографии
|"c3d4e5f6-a7b8-4c9d-8e0f-1a2b3c4d5e6f"

|listing_id
|UUID
|FOREIGN KEY → `listings(id)` ON DELETE CASCADE, NOT NULL
|Идентификатор объявления, к которому привязано фото
|"5a7f3c8e-1b2d-4e5f-8a9b-0c1d2e3f4a5b"

|photo_url
|TEXT
|NOT NULL
|URL фотографии в MinIO (внутренний; в API-ответах переписывается на относительный путь `/minio/...`)
|"http://minio:9000/brandhunt-photos/5a7f.../0.jpg"

|sort_order
|SMALLINT
|NOT NULL, DEFAULT 0
|Порядковый номер фото в галерее объявления (0 — первое фото поста)
|0, 1, 2

|is_cover
|BOOLEAN
|NOT NULL, DEFAULT FALSE
|Признак обложки (титульное фото, отображаемое в списке объявлений). В рамках одного объявления обложка одна — обычно это фото с `sort_order = 0`
|true, false

|uploaded_at
|TIMESTAMP WITH TIME ZONE
|NOT NULL, DEFAULT NOW()
|Дата загрузки в MinIO
|2026-04-17 12:05:00+00
|===

На таблицу наложено ограничение уникальности: `UNIQUE (listing_id, sort_order)` — у одного объявления не может быть двух фото с одинаковой позицией в галерее.

[[favorites]]
=== 5. Избранное
Наименование: `favorites` +
Описание: Таблица избранных объявлений пользователей. Связь «многие ко многим» между `users` и `listings`
[cols="18%, 15%, 15%, 25%, 27%"]
|===
|Параметр |Тип данных |Доп. ограничения |Описание |Примеры

|id
|UUID
|PRIMARY KEY, NOT NULL, DEFAULT gen_random_uuid()
|Идентификатор записи в избранном
|"d4e5f6a7-b8c9-4d0e-9f1a-2b3c4d5e6f7a"

|user_id
|UUID
|FOREIGN KEY → `users(id)` ON DELETE CASCADE, NOT NULL
|Идентификатор пользователя
|"5a7f3c8e-1b2d-4e5f-8a9b-0c1d2e3f4a5b"

|listing_id
|UUID
|FOREIGN KEY → `listings(id)` ON DELETE CASCADE, NOT NULL
|Идентификатор объявления
|"b2c3d4e5-a7f8-4b9c-8d0e-1f2a3b4c5d6e"

|saved_at
|TIMESTAMP WITH TIME ZONE
|NOT NULL, DEFAULT NOW()
|Дата добавления объявления в избранное
|2026-04-17 12:05:00+00
|===

На таблицу наложено ограничение уникальности: `UNIQUE (user_id, listing_id)` — одно и то же объявление нельзя добавить в избранное дважды; эндпоинт «добавить в избранное» становится идемпотентным.

[[monitoring_log]]
=== 6. Лог обработки сообщений
Наименование: `monitoring_log` +
Описание: Технический лог работы мониторов. Записывается строка на каждое обработанное сообщение из TG/VK. Используется админкой для диагностики и подсчёта статистики
[cols="18%, 15%, 15%, 25%, 27%"]
|===
|Параметр |Тип данных |Доп. ограничения |Описание |Примеры

|id
|BIGSERIAL
|PRIMARY KEY, NOT NULL
|Автоинкрементный идентификатор записи. `BIGSERIAL` выбран вместо `UUID` из-за ожидаемо большого объёма вставок
|1, 2, 3, ...

|source_id
|UUID
|FOREIGN KEY → `sources(id)` ON DELETE SET NULL
|Идентификатор источника, в контексте которого обрабатывалось сообщение. Может быть `NULL`, если источник удалён
|"b2c3d4e5-a7f8-4b9c-8d0e-1f2a3b4c5d6e"

|message_id
|VARCHAR(255)
|—
|Идентификатор сообщения во внешней системе (post_id для VK, message_id для TG)
|"12345"

|status
|VARCHAR(20)
|NOT NULL
|Итог обработки сообщения. Только одно значение
|"parsed", "skipped", "error"

|error_msg
|TEXT
|—
|Текст ошибки (заполняется только при `status = "error"`)
|"ollama: connection refused"

|processed_at
|TIMESTAMP WITH TIME ZONE
|NOT NULL, DEFAULT NOW()
|Время обработки сообщения
|2026-04-17 12:05:00+00
|===

**Значения поля `status`:**

* `parsed` — сообщение распознано LLM как объявление, атрибуты извлечены, записи созданы в `listings`
* `skipped` — сообщение не является объявлением (по оценке LLM), записи в `listings` не создавались
* `error` — произошла ошибка при обращении к LLM или при сохранении в БД/MinIO; текст ошибки в `error_msg`
